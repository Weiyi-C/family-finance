# 家庭记账系统 — 项目总结文档

> 版本：v1.0
> 日期：2026-07-15
> 状态：设计完成，待开发

---

## 一、项目概述

### 1.1 项目背景

市面主流记账App存在以下痛点：
- **钱迹**：手动记账体验好，但无AI能力，不支持家庭协作
- **鲨鱼记账**：极简快速，但分类粗糙，有广告
- **咔皮记账**：AI能力强，但高级功能收费，手动体验一般
- **随手记**：功能全面，但复杂臃肿，学习成本高
- **乖猫记账**：聊天式记账有趣，但多账本无法合并分析

没有一款App能同时满足：**快速记账 + 完整资金追踪 + 家庭协作 + 灵活分析 + 数据自主**。

### 1.2 项目目标

打造一个**家庭级记账系统**，实现：
- 📱 手机端快速记账（离线可用）
- 💻 网页端后台管理 + 数据大屏
- 🏠 家庭成员协作，账单汇总
- 🤖 AI辅助分类、识别、分析（OCR在客户端完成）
- 🔒 数据完全自主，部署在家里

### 1.3 目标用户

| 用户 | 角色 | 主要场景 |
|------|------|---------|
| 本人 | owner（创建者） | 日常记账、导入账单、数据分析、系统管理 |
| 女朋友 | admin（管理员） | 快速记账、查看家庭收支、导入自己的账单 |

## 二、需求分析

### 2.1 核心需求（P0）

#### 2.1.1 快速记账

**目标：3秒完成一笔记录**

```
用户场景：午饭后想快速记一笔
期望操作：打开App → 输金额 → 选分类 → 完成
容许操作：打开App → 输金额 → 选分类 → 选账户 → 完成
```

**设计要点：**
- 大金额输入框，数字键盘自动弹出
- 常用分类置顶，支持搜索
- 支持"快速录入"模式：只填金额+分类，其他信息后续补充
- 桌面小组件直接进入记账页

#### 2.1.2 完整资金追踪

**三维账单模型：这笔钱从哪来、怎么付的、在哪花的**

```
场景：女朋友用支付宝亲密付在淘宝买了一条裙子，扣的是我的花呗

记录：
├── 资金来源：张三的花呗（钱从哪来）
├── 支付渠道：支付宝（通过什么付的）
├── 交易平台：淘宝（在哪消费的）
├── 记录人：女朋友（谁记的账）
└── 付款人：张三（谁实际付的钱）
```

**为什么需要三维？**
- 只记"支付宝付了299" → 不知道是花呗还是余额还是银行卡
- 只记"淘宝买东西" → 不知道是支付宝付的还是微信付的
- 三个维度独立记录，才能完整还原消费场景

#### 2.1.3 家庭协作

```
家庭组：
├── owner（创建者）：所有功能 + 删除家庭 + 重置邀请码
├── admin（管理员）：记账、导入、导出、管理分类/预算/规则
├── member（成员）：记账、查看汇总（可通过 families.settings 开放更多权限）
└── 协作方式：
    ├── 每人记自己的
    ├── 可以帮别人记（paid_by字段）
    └── admin 及以上可查看/编辑所有记录
```

#### 2.1.4 灵活分析

**支持任意时间维度和分析角度：**

```
时间维度：
├── 按天/周/月/季/年
└── 自定义起止日期

分析维度：
├── 总收支概览（收入 vs 支出 vs 净额）
├── 分类占比（饼图/环形图）
├── 趋势折线图（日/周/月趋势）
├── 同比/环比（本月 vs 上月）
├── 支出排行（TOP N 分类/商户）
├── 个人 vs 家庭对比
└── 三维交叉分析（资金来源 × 支付渠道 × 交易平台）
```

### 2.2 重要需求（P1）

#### 2.2.1 账单导入

**支持多种来源，自动解析：**

| 来源 | 格式 | 解析器 |
|------|------|--------|
| 支付宝 | CSV（GBK编码） | AlipayParser |
| 微信支付 | CSV（UTF-8，多余表头） | WechatParser |
| 招商银行 | CSV/Excel | CmbParser |
| 工商银行 | CSV/Excel | IcbcParser |
| 京东 | CSV | JingdongParser |
| 通用模板 | Excel/CSV | TemplateParser |

**导入流程：**
```
上传文件 → 自动识别平台 → 解析提取 → 智能匹配已有记录 → 用户确认 → 批量入库
```

**智能匹配：**
- 导入的账单自动匹配之前快速录入的记录
- 匹配规则：金额相同 + 时间相差<24h + 同用户
- 匹配到 → 补充完善信息
- 未匹配 → 新建记录

#### 2.2.2 拍照/截图识别

**三种识别场景：**

| 场景 | 输入 | 提取内容 |
|------|------|---------|
| 小票/发票 | 线下消费凭证 | 商户名、金额、商品明细 |
| 支付宝截图 | 账单详情页截图 | 交易时间、对方、金额、支付方式 |
| 微信截图 | 账单详情页截图 | 交易时间、对方、金额 |

**识别方案（全部在客户端完成，不占用服务器资源）：**
- 方案A（推荐）：设备端OCR（Flutter ML Kit）+ 后端AI结构化提取
- 方案B（最准）：客户端直调多模态AI看图提取
- 方案C（最快）：客户端直调云端OCR服务（百度/腾讯）

#### 2.2.3 预算管理

**支持多维度预算：**

```
预算类型：
├── 全局月度预算：总支出不超过 15000
├── 分类月度预算：餐饮 3000、交通 800
├── 单账本预算：旅行账本预算 8000
└── 按周预算：每周餐饮不超过 800

展示方式：
├── 进度条（已花/总预算）
├── 颜色渐变（绿→黄→红）
├── 剩余可用天均额
└── 超支预警推送
```

#### 2.2.4 多币种支持

**出国旅行记账，三阶段汇率：**

```
7月10日：东京便利店消费 ¥500 JPY
├── 记账时：按当天市场汇率 0.048 → 24.00 CNY
├── 银行入账：按银行汇率 0.0482 → 24.10 CNY
└── 差额：0.10 元（汇率波动差异）

三种统计口径：
├── 记账金额：旅行中实时看花了多少
├── 结算金额：最终精确对账
└── 原始币种：看外币总花了多少
```

### 2.3 增强需求（P2）

#### 2.3.1 AI智能功能

```
AI能力矩阵：
├── 自然语言记账："打车25" → {分类:交通, 金额:25}
├── 智能分类：关键词规则 + AI兜底，准确率90%+
├── 自动标签：分类的同时自动打标签
├── 分类学习：用户修正后自动记住
├── 截图识别：小票/账单截图自动提取信息
├── 收支分析：月度报告、消费洞察
└── 智能建议："本月外卖占比60%，建议控制"
```

#### 2.3.2 借贷管理

```
场景：借给同事小王500元
├── 记录：type=lend, counterparty="小王", amount=500
├── 状态追踪：pending → partial → settled
├── 还款记录：每笔还款明细
└── 统计："还有多少借出去的钱没收回来？"
```

#### 2.3.3 退款追踪

```
场景：淘宝退货退款299元
├── 原始消费：type=expense, amount=299
├── 退款申请：transaction_refunds 记录 status='pending'
├── 退款到账：新建 type=income 的交易 + 更新退款状态为 'received'
└── 统计："本月有多少退款待到账？"
```

#### 2.3.4 报销管理

```
流程：
1. 日常消费时标记 #可报销
2. 月底汇总可报销消费
3. 创建报销单，关联消费记录
4. 提交 → 审批 → 到账
5. 统计："还有多少报销款没收回来？"
```

#### 2.3.5 储蓄目标

```
场景：存钱去日本旅行，目标10000元
├── 设定目标：name="日本旅行", target=10000
├── 每月存入：current_amount += 2000
├── 进度展示："已存 6000/10000 (60%)"
└── 达成提醒："恭喜达成目标！"
```

### 3.2 技术选型

| 层级 | 技术 | 理由 |
|------|------|------|
| 手机端 | Flutter (Dart) | 语法接近Java，一套代码出iOS+Android |
| 网页端 | Vue3 + Element Plus | 学习曲线平缓，组件丰富 |
| 后端API | Python FastAPI | 开发快，AI集成方便 |
| 数据库 | PostgreSQL | 支持JSONB，多端同步 |
| 文件存储 | 服务器本地磁盘 | 家庭使用量小，无需对象存储；OCR在客户端完成 |
| AI | 大模型API | 智能分类、账单识别、收支分析 |
| 部署 | Docker Compose | 一键部署，适合家庭服务器 |

### 3.3 离线同步架构

```

│ 2. 查账          │              │ 2. 处理导入      │

│ 4. 离线完全可用  │              │ 4. 数据备份      │
└──────────────────┘              └──────────────────┘

同步策略：
├── 增量同步：只传变化的记录（基于逻辑序列号）
├── 冲突处理：以服务器为准
├── 双向同步：手机上传新记录 + 下载服务器更新
└── 自动触发：启动时、网络恢复时、定时、记账后
```

## 四、数据库设计

### 4.1 核心数据模型

```
43张服务器端表 + 10张手机端表

核心表：
├── families          — 家庭组
├── users             — 用户
├── account_books     — 账本（可分可合）
├── transactions      — 交易记录（双式记账）
├── categories        — 三级分类（24+106+55）
├── payment_accounts  — 资金账户（模板+实例）
├── payment_channels  — 支付渠道
├── platforms         — 交易平台
└── budgets           — 预算

扩展表：
├── currencies + exchange_rates    — 多币种与汇率
├── recurring_transactions         — 周期性交易
├── bill_imports + bill_import_items — 账单导入
├── automation_rules               — 自动分类+标签
├── attachments                    — 附件（客户端OCR后上传结果）
├── tags + transaction_tags        — 标签
├── debts + debt_repayments        — 借贷管理
├── reimbursements                 — 报销管理
├── savings_goals                  — 储蓄目标
├── merchant_aliases               — 商户别名
├── notifications                  — 通知
├── user_settings                  — 用户设置
├── family_sync_seq + sync_change_log + client_sync_state — 离线同步(变更日志+逻辑序列号)
├── backup_configs + backup_logs   — 自动备份
├── account_balance_snapshots      — 账户余额月度快照(性能优化)
├── credit_card_bills              — 信用卡账单管理(自动生成+还款提醒)
└── app_error_logs + app_slow_queries — 应用监控(错误日志+慢查询)
```

### 4.2 关键设计决策

| 决策 | 方案 | 理由 |
|------|------|------|
| 记账方式 | 双式记账法 | 账户余额自动计算，永不手动调整 |
| 账本设计 | 逻辑分组，非物理隔离 | 支持单账本/多账本/全局任意分析 |
| 账单模型 | 资金来源×支付渠道×交易平台 | 完整还原亲密付等复杂场景 |
| 快速录入 | 先记最简，后续完善 | 兼顾记账速度和数据完整性 |
| 金额存储 | 全链路 BIGINT（单位：分），显示时÷100 | 彻底避免浮点精度问题，手机/服务器/同步零转换 |
| 分类体系 | 三级，系统预设+家庭自定义 | 覆盖全面且可扩展 |
| 离线策略 | 手机本地SQLite，有网同步 | 无网也能用，数据不丢失 |
| 余额计算 | 月度快照 + 增量SUM | 1核2G上也能毫秒级查询余额 |
| 预算结转 | rollover字段控制 | 未用完预算可选结转到下期 |
| 大表优化 | 按季度分区 + 超2年归档 | 数据量增长后查询性能不退化 |
| 全文搜索 | pg_trgm + GIN索引（开箱即用），可选zhparser中文分词 | 商户名/备注模糊搜索，不走全表扫描 |
| 分类归档 | 归档前检查引用 | 防止归档被交易/规则/预算引用的分类 |

### 4.3 约束与索引

```
CHECK约束（13个）：
├── entry_side IN ('debit', 'credit')
├── type IN ('expense', 'income', 'transfer')
├── amount > 0
├── chk_completion ('complete', 'pending', 'partial')
├── chk_refund_status ('pending', 'approved', 'received') -- transaction_refunds表
├── chk_settle_status ('pending', 'settled')               -- transaction_settlements表
├── category level IN (1, 2, 3)
├── category type IN ('expense', 'income')
├── debt type IN ('lend', 'borrow')
├── debt status IN ('pending', 'partial', 'settled')
├── reimbursement status IN ('draft', 'submitted', 'approved', 'received')
├── savings goal status IN ('active', 'achieved', 'abandoned')
└── ...

索引（27个）：
├── transactions: family_id, book_id, time, category, entry_id,
│                 payment_account, recorded_by, paid_by, import_id, recurring_id
├── 其他表：按查询场景建立索引
└── GIN索引：JSONB字段（raw_data, ocr_result, conditions, actions）
```

### 6.1 性能要求

| 指标 | 目标 |
|------|------|

### 6.2 安全要求

| 措施 | 说明 |
|------|------|
| 密码加密 | bcrypt哈希存储 |

| 角色权限 | owner > admin > member 三级角色 + families.settings 功能开关 |
| 数据传输 | HTTPS加密 |
| 数据备份 | 每天自动备份 |
| 数据自主 | 部署在家里，不出门 |

| 平台 | 要求 |
|------|------|
| iOS | 14.0+ |
| Android | 8.0+ |
| 网页 | Chrome/Edge/Safari 最新版 |
| 服务器 | Docker支持的任何Linux |

## 八、风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| AI识别准确率不够 | 分类出错 | 关键词规则兜底 + 用户修正学习 |
| 离线同步冲突 | 数据不一致 | 以服务器为准 + 冲突日志 |
| 服务器宕机 | 手机端仍可用 | 离线优先架构，服务器恢复后自动同步 |
| 账单格式变化 | 导入失败 | 解析器可扩展 + 通用模板兜底 |
| 数据丢失 | 灾难性 | 三层备份（本地+远程+手机） |

## 附录B：术语表

| 术语 | 说明 |
|------|------|
| 双式记账 | 每笔交易记录资金来源和去向，借贷平衡 |
| 账本 | 逻辑分组标签，如日常、旅行、装修 |
| 三维模型 | 资金来源 × 支付渠道 × 交易平台 |
| 快速录入 | 只填金额+分类的极简记账模式 |
| 延迟完善 | 先快速记录，后续导入账单补充信息 |
| 离线优先 | 手机本地存储优先，有网时同步 |
| 规则引擎 | 根据关键词自动分类+打标签 |
| 信封预算法 | 给每一块钱安排去处的预算方法 |
