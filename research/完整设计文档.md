## 一、系统概述

### 1.1 项目目标

打造一个家庭级记账系统，实现：
- 📱 手机端 3 秒快速记账，离线可用
- 💻 网页端后台管理 + 数据大屏
- 🏠 家庭成员协作，账单汇总
- 🤖 AI 辅助分类、识别、分析
- 🔒 数据完全自主，部署在自己的服务器

### 1.3 用户角色

三级角色，由高到低：owner > admin > member

| 角色 | 权限 | 场景 |
|------|------|------|
| owner（创建者） | 所有功能 + 删除家庭 + 重置邀请码 | 本人，唯一的家庭拥有者 |
| admin（管理员） | 记账、导入、导出、管理分类/预算/规则、移除成员 | 女朋友，信任的协作者 |
| member（普通成员） | 记账、查看家庭汇总、查看统计 | 未来扩展（父母等） |

权限控制方式：预设角色 + families.settings JSONB 功能开关
- member 默认只能记账和查看，可通过开关开放导入、导出、管理预算等
- owner 和 admin 的差异仅在于"删除家庭"和"移除 owner"

### 3.2 核心表结构

#### transactions（交易记录，最核心）

```sql
CREATE TABLE transactions (
    id                  BIGSERIAL PRIMARY KEY,
    family_id           BIGINT NOT NULL REFERENCES families(id),
    book_id             BIGINT NOT NULL REFERENCES account_books(id),

    -- 双式记账
    entry_id            BIGINT,                     -- 同一笔交易的借贷共享
    entry_side          VARCHAR(4) NOT NULL,        -- debit / credit

    -- 基础
    type                VARCHAR(20) NOT NULL,       -- expense / income / transfer
    amount              BIGINT NOT NULL,     -- 金额（正数）

    -- 多币种（仅外币交易时填写）
    currency            VARCHAR(3) DEFAULT 'CNY',
    original_amount     BIGINT,              -- 原始外币金额
    original_currency   VARCHAR(3),
    exchange_rate       DECIMAL(10,6),              -- 记账时汇率
    -- 外币结算信息拆分到 transaction_settlements 表
    -- 退款追踪拆分到 transaction_refunds 表

    -- 三级分类
    category_id         BIGINT REFERENCES categories(id),
    sub_category_id     BIGINT REFERENCES categories(id),
    detail_category_id  BIGINT REFERENCES categories(id),

    -- 三维模型
    payment_account_id  BIGINT REFERENCES payment_accounts(id),
    payment_channel_id  BIGINT REFERENCES payment_channels(id),
    platform_id         BIGINT REFERENCES platforms(id),

    -- 商户
    merchant_name       VARCHAR(200),
    description         VARCHAR(500),

    -- 时间
    transaction_time    TIMESTAMP NOT NULL,
    recorded_at         TIMESTAMP DEFAULT NOW(),

    -- 人
    recorded_by         BIGINT NOT NULL REFERENCES users(id),
    paid_by             BIGINT REFERENCES users(id),

    -- 快速录入
    is_quick_entry      BOOLEAN DEFAULT FALSE,
    completion_status   VARCHAR(20) DEFAULT 'complete',

    -- 关联
    recurring_id        BIGINT REFERENCES recurring_transactions(id),
    import_id           BIGINT REFERENCES bill_imports(id),
    raw_data            JSONB,

    -- 同步字段
    version             INTEGER NOT NULL DEFAULT 1,
    is_deleted          BOOLEAN DEFAULT FALSE,
    created_at          TIMESTAMP DEFAULT NOW(),
    updated_at          TIMESTAMP DEFAULT NOW(),

    -- 约束
    CONSTRAINT chk_entry_side CHECK (entry_side IN ('debit', 'credit')),
    CONSTRAINT chk_type CHECK (type IN ('expense', 'income', 'transfer')),
    CONSTRAINT chk_amount CHECK (amount > 0),
    CONSTRAINT chk_completion CHECK (completion_status IN ('complete', 'pending', 'partial'))
);

-- 索引 (11个)
CREATE INDEX idx_txn_family ON transactions(family_id);
CREATE INDEX idx_txn_book ON transactions(book_id);
CREATE INDEX idx_txn_time ON transactions(transaction_time);
CREATE INDEX idx_txn_category ON transactions(category_id);
CREATE INDEX idx_txn_entry ON transactions(entry_id);
CREATE INDEX idx_txn_account ON transactions(payment_account_id);
CREATE INDEX idx_txn_recorded_by ON transactions(recorded_by);
CREATE INDEX idx_txn_paid_by ON transactions(paid_by);
CREATE INDEX idx_txn_import ON transactions(import_id);
CREATE INDEX idx_txn_recurring ON transactions(recurring_id);
```

### 3.3 双式记账原理

```
用户操作：用支付宝余额买午饭 35 元
系统自动产生两条记录：

┌─────────────────────────────────────────────────────┐
│ 记录 A（借方）                                       │
│ id: 1001, entry_id: 500, entry_side: 'debit'        │
│ type: 'expense', amount: 3500                        │
│ category_id: 餐饮→午饭                               │
│ payment_account_id: 支付宝余额                       │
│ → 含义：钱花在了餐饮上                               │
├─────────────────────────────────────────────────────┤
│ 记录 B（贷方）                                       │
│ id: 1002, entry_id: 500, entry_side: 'credit'       │
│ type: 'expense', amount: 3500                        │
│ payment_account_id: 支付宝余额                       │
│ → 含义：钱从支付宝余额扣除                           │
└─────────────────────────────────────────────────────┘

用户只看到一条 "午饭 35 元"
账户余额 = initial_balance + SUM(借方收入) - SUM(贷方支出) + SUM(借方转入) - SUM(贷方转出)

转账示例：支付宝 → 银行卡 1000 元
  借方：银行卡 +1000（钱到哪了）
  贷方：支付宝 -1000（钱从哪来）
  总资产不变
```

## 七、核心机制设计

### 7.1 离线同步机制（基于变更日志 + 逻辑序列号）

```
手机端 (SQLite)                     服务器 (PostgreSQL)
┌──────────────────┐               ┌──────────────────┐
│ 本地数据库        │               │ 主数据库          │
│                  │   有网时      │                  │
│ 正常记账         │ ──同步──→    │ sync_change_log  │
│ 查账、看报表     │ ←──同步──    │ (逻辑序列号)      │
│ 离线完全可用     │               │                  │
└──────────────────┘               └──────────────────┘

同步协议:
1. PUSH: 客户端上传变更 → 服务器按版本号检测冲突 → 原子写入
2. PULL: 客户端带 last_seq 拉取 → 服务器返回 seq 之后的所有变更日志

冲突处理: base_version == DB.version → 无冲突; 否则返回服务器最新数据
双式记账: debit/credit 两条记录在事务中原子写入
同步时机: 启动时、网络恢复时、定时5分钟、记账后延迟3秒
```

### 7.2 账单导入机制

```
上传文件
  │
  ▼
自动识别平台（文件名+表头+编码检测）
  │
  ▼
调用对应解析器（AlipayParser / WechatParser / ...）
  │
  ▼
统一中间格式 ImportItem
  │
  ▼
智能匹配已有快速记录（金额+时间+用户）
  │
  ├── 匹配到 → 合并（补充信息）
  └── 未匹配 → 新建记录
  │
  ▼
AI 辅助分类（关键词规则 → AI 兜底）
  │
  ▼
用户确认 → 批量入库
```

### 7.3 自动分类规则引擎

```
一笔新交易进入系统
  │
  ▼
阶段 1：自动分类（关键词匹配 → AI 兜底）
  │
  ▼
阶段 2：自动标签（可命中多个标签）
  │
  ▼
阶段 3：数据修正（退款类型修改等）
  │
  ▼
用户看到：分类 + 标签 + 平台

用户修正 → 自动学习为新规则
```

### 7.4 自动备份机制

```
每天凌晨 3:00：
  ├── pg_dump 全量备份 → 压缩 → 本地存储
  └── 推送到配置的远程目标（邮箱/OSS/NAS）
```

```
认证方案：短期 Access Token + 长期 Refresh Token（无需 Redis）
  ├── Access Token: 有效期 15 分钟，每次请求携带
  ├── Refresh Token: 有效期 30 天，存数据库 refresh_tokens 表
  ├── Token 过期 → 用 Refresh Token 换新 Access Token
  ├── 登出 → 删除 refresh_tokens 记录，即刻失效
  └── 每次请求携带 Authorization: Bearer <access_token>

权限控制：三级角色 + 功能开关
  ├── owner: 所有功能 + 删除家庭 + 重置邀请码
  ├── admin: 记账、导入、导出、管理分类/预算/规则、移除成员
  ├── member: 默认记账 + 查看汇总 + 查看统计
  │   └── 可通过 families.settings 开关开放更多功能
  │       ├── member_can_import: false
  │       ├── member_can_export: false
  │       ├── member_can_delete_others: false
  │       ├── member_can_view_all_accounts: true
  │       ├── member_can_manage_budget: false
  │       └── member_can_manage_categories: false
  └── 数据隔离：family_id 强制过滤，跨家庭数据不可见

API 权限检查示例：
  POST /api/transactions          → require_role("member")
  POST /api/import/upload         → require_role("admin")
  DELETE /api/families/{id}       → require_role("owner")
```

### 8.2 数据安全

```
密码存储：bcrypt 哈希（不可逆）
数据传输：HTTPS 加密
SQL 注入：ORM 参数化查询
XSS 防护：前端输出转义
数据备份：每日自动备份
敏感数据：API Key 等存环境变量
```

### 8.3 API 安全

```
请求频率限制：60 次/分钟
CORS 配置：只允许指定域名
输入验证：Pydantic 严格校验
错误处理：不暴露内部信息
```

### 8.4 应用层监控（轻量方案，无需额外服务）

**设计原则：** 在 1核2G 服务器上不能跑额外的监控服务（Prometheus/Grafana 等太重），采用应用内置的轻量方案。

```
三层监控：
├── 1. 结构化日志（JSON 格式，stdout 输出）
├── 2. 健康检查端点（Docker + 外部探测）
└── 3. 错误与慢查询记录（写入 PostgreSQL）
```

#### 8.4.1 结构化日志

```python
# 使用 structlog，输出 JSON 到 stdout，Docker 自动收集
# pip install structlog

import structlog
import time

structlog.configure(
    processors=[
        structlog.processors.TimeStamper(fmt="iso"),
        structlog.processors.add_log_level,
        structlog.processors.JSONRenderer()
    ]
)
logger = structlog.get_logger()

# 请求日志中间件
@app.middleware("http")
async def logging_middleware(request: Request, call_next):
    start = time.monotonic()
    response = await call_next(request)
    duration_ms = (time.monotonic() - start) * 1000

    logger.info("request",
        method=request.method,
        path=request.url.path,
        status=response.status_code,
        duration_ms=round(duration_ms, 1),
        user_id=getattr(request.state, "user_id", None),
        client_ip=request.client.host
    )

    # 慢请求告警（>1秒）
    if duration_ms > 1000:
        logger.warning("slow_request",
            path=request.url.path,
            duration_ms=round(duration_ms, 1)
        )

    return response
```

```python
async def health_check():
    checks = {}

        seq = await db.fetchval(

    all_ok = all(v == "ok" for v in checks.values())

    )
```

```python
# 全局异常处理器
@app.exception_handler(Exception)
async def global_exception_handler(request: Request, exc: Exception):
    logger.error("unhandled_exception",
        endpoint=request.url.path,
        error_type=type(exc).__name__,
        message=str(exc),
        traceback=traceback.format_exc()[:2000]
    )

    # 写入数据库（异步，不阻塞响应）
    await db.execute("""
        INSERT INTO app_error_logs (level, endpoint, method, user_id, error_type, message, traceback)
        VALUES ('error', $1, $2, $3, $4, $5, $6)
    """, request.url.path, request.method,
         getattr(request.state, "user_id", None),
         type(exc).__name__, str(exc)[:500],
         traceback.format_exc()[:2000])

    return JSONResponse(
        status_code=500,
        content={"detail": "服务器内部错误"}
    )
```
